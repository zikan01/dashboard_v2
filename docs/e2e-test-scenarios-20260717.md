## ① 로그인·네비게이션

**경로**: `/login, / (전역 사이드바·탑바)`

미로그인 리다이렉트(AuthGuard), 이메일/비밀번호 로그인 성공·실패, 사이드바 메뉴 구성(문자 발송 그룹 + 옵션 업로드 등 merge 공존), 탑바 제목/버튼, 로그아웃, staff 권한 게이트(RoleGate)를 브라우저에서 사람이 직접 검증한다.

**사전 조건**
- dev 서버 http://localhost:3000 실행 중이고 Supabase(서울) 접속 가능
- owner 계정 contact@itconnect.dev 가 Supabase Auth에 존재하고 profiles.status='active', role='owner'이며 비밀번호를 알고 있음
- staff 시나리오(시나리오 7)는 profiles.status='active', role='staff'인 직원 계정이 필요 — 없으면 owner로 /staff(직원 관리)에서 초대 후 진행
- 비활성 계정 엣지 케이스는 profiles.status가 'active'가 아닌 계정이 필요 (없으면 해당 케이스 생략)
- 미로그인 시나리오는 시크릿 창 또는 로그아웃 상태에서 시작

**필요 리소스·환경변수**: `dev 서버 http://localhost:3000 (Next.js 14)`, `Supabase 서울 프로젝트 — Auth(이메일/비밀번호) + profiles 테이블(business_id, name, role, status)`, `owner 계정 contact@itconnect.dev 의 비밀번호`, `staff 시나리오용 활성 직원 계정 (owner가 /staff 화면에서 초대 가능)`, `비활성 계정 케이스용 status≠active 계정 (선택)`

## ② 홈·달력·내보내기 스모크

**경로**: `/ , /calendar , /export`

대시보드 홈, 예약 캘린더, 엑셀 내보내기 세 화면이 정상 로드되고 핵심 위젯(KPI·캘린더 격자·필터·내보내기 버튼)이 코드에 정의된 라벨·문구대로 표시·동작하는지 가벼운 스모크 수준으로 확인한다.

**사전 조건**
- dev 서버 http://localhost:3000 실행 중
- contact@itconnect.dev (owner 권한, 사업장 "고마워할매") 계정으로 로그인된 브라우저 세션
- 예약 데이터 존재 여부는 불명 — 데이터가 필요한 시나리오는 각 시나리오 사전 조건에 별도 명시했고, 데이터가 없으면 빈 상태 문구 검증으로 대체 가능
- 세 화면 모두 클라이언트 컴포넌트로 useData()의 ready가 false인 동안 null을 렌더하므로, 접속 직후 잠깐 빈 화면이 보이는 것은 정상

## ④ 예약 목록·상세

**경로**: `/reservations, /reservations/[id]`

## ⑤ 옵션·준비물 (옵션 업로드 화면 + 예약 상세 준비물 카드)

**경로**: `/options (예약 상세 매칭 확인은 /reservations/[id])`

옵션별 준비물 CRUD(등록·병합·수정·삭제), 옵션명 부분 일치 매칭(공백·대소문자 무시, 최장 키워드 우선, 다중 매칭 합집합), 중복 등록 방지(409 병합 모달), 목록 가나다순 정렬, owner 전용 권한 게이트를 검증한다.

**사전 조건**
- contact@itconnect.dev (owner) 계정으로 http://localhost:3000 로그인 상태
- dev 서버 실행 중, Supabase 마이그레이션 적용됨 (preparation_items 테이블 존재)
- 매칭·미등록 카드 시나리오(5·6·7)는 옵션 텍스트가 포함된 예약 데이터가 최소 1건 필요 — 없으면 예약을 먼저 생성/수집해 둘 것
- 권한 시나리오(8)는 role=staff 직원 계정이 필요 — 존재 여부 불명이므로 없으면 생성 필요
- 테스트 시작 전 등록된 옵션 목록 상태를 기록해 두고, 테스트 후 생성한 키워드(바베큐·계곡·계곡체험·가마솥 등)를 삭제해 원복할 것

**필요 리소스·환경변수**: `Supabase (서울) — preparation_items 테이블, business_id 스코프`, `로그인 세션 (requireUser) — GET은 로그인 사용자 전체, POST/PATCH/DELETE는 owner만`, `예약 데이터 (reservations.options) — Tier 2 미등록 옵션 카드와 예약 상세 매칭 표시에 필요`

## ⑥ 자동 안내 설정

**경로**: `/settings/notifications (API: PUT /api/settings/notifications)`

대표(owner) 전용 자동 안내 설정 화면 검증 — 전체 활성화 스위치, D-7·D-3·D-1·당일 규칙의 on/off·발송 시각·템플릿 지정, 실시간 미리보기, 저장 성공/실패 피드백 및 서버 측 검증(템플릿 미지정 활성화 거부, Zod 입력 검증, owner 권한 게이트)

**사전 조건**
- contact@itconnect.dev (owner 권한) 계정으로 로그인 가능해야 함
- notification_rules 테이블에 해당 사업장(고마워할매)의 d_7·d_3·d_1·d_day 4개 행이 존재해야 함 — 없으면 규칙 카드가 하나도 렌더링되지 않음 (페이지는 DB에서 규칙 목록을 직접 조회)
- message_templates에 is_active=true인 템플릿이 1개 이상 존재해야 함 (템플릿 선택·미리보기·활성화 저장 시나리오에 필요)
- 권한 부정 케이스용으로 staff 권한 계정이 있으면 좋음 — 없으면 해당 엣지 케이스는 API 직접 호출(비로그인)로 대체
- 예약 데이터는 불필요 — 이 화면은 설정 테이블만 읽고 씀

**필요 리소스·환경변수**: `dev 서버 http://localhost:3000 실행 중`, `Supabase 연결 (business_notification_settings, notification_rules, message_templates, system_audit_logs 테이블)`, `NOTIFICATION_SEND_MODE는 이 화면에 영향 없음 — 설정 저장만 하며 실제 발송을 트리거하지 않음`

## ⑧ SOLAPI 설정

**경로**: `/settings/providers/solapi`

SOLAPI 연결 상태(키 등록 여부·잔액·발신번호·발송 모드) 표시와 allowlist 기반 테스트 문자 발송 기능 검증

**사전 조건**
- dev 서버 http://localhost:3000 실행 중
- owner 계정(contact@itconnect.dev)으로 로그인
- SOLAPI_API_KEY·SOLAPI_API_SECRET·SOLAPI_SENDER_NUMBER 환경변수 설정됨

- NOTIFICATION_TEST_PHONE_ALLOWLIST 비어있음 (테스트 발송 성공 시나리오는 별도 사전 조건 필요)

**필요 리소스·환경변수**: `SOLAPI_API_KEY`, `SOLAPI_API_SECRET`, `SOLAPI_SENDER_NUMBER`, `NOTIFICATION_SEND_MODE`, `NOTIFICATION_TEST_PHONE_ALLOWLIST`

### 8.1 [P0] 설정 상태 카드 표시 — 키 등록·연결·잔액·발신번호·발송 모드

## ⑨ 발송 일정

**경로**: `/notifications/schedule`

발송 예정 작업의 탭별 목록(오늘·이번 주·전체 예정·취소·제외) 조회와 대표(owner) 전용 행 액션 — 즉시 발송(dry_run 단건 디스패치), 일회성 제외, 발송 시각 변경 — 및 경합(동시 조작) 가드·권한 게이트를 검증한다.

**사전 조건**
- contact@itconnect.dev (owner 권한) 계정으로 로그인되어 있을 것
- notification_jobs 테이블에 status='scheduled'(화면 표시 '발송 예정')인 작업이 최소 2건 존재할 것 — 예약 데이터 존재 여부 불명이므로, 없으면 방문일이 미래인 예약을 생성해 D-7/D-3/D-1/당일 잡이 자동 생성되도록 준비
- NOTIFICATION_SEND_MODE=dry_run 상태 유지 (지금 발송 시 실제 문자 발송 없이 드라이런으로 집계됨)
- 직원 권한 시나리오(P2)는 role이 owner가 아닌 별도 staff 계정 필요 — 없으면 해당 시나리오 생략
- 경합 가드 시나리오는 같은 계정으로 브라우저 탭 2개를 띄울 수 있어야 함

**필요 리소스·환경변수**: `NOTIFICATION_SEND_MODE=dry_run (send_now 시 mock provider 사용, 드라이런 집계)`, `NOTIFICATION_TEST_PHONE_ALLOWLIST 비어있음 (allowlist 필터 미적용)`, `PHONE_HASH_PEPPER 설정됨 (dispatcher가 없으면 예외 발생)`, `Supabase (notification_jobs·business_notification_settings·system_audit_logs 접근)`, `CRON_SECRET (크론 디스패처와의 경합 시나리오 배경)`

### 9.6 [P1] 경합 가드 — 오래된 화면에서의 액션 충돌 (409)

**절차**
1. 같은 계정으로 브라우저 탭 A·B 두 개에 발송 일정 페이지를 각각 열기
2. 탭 A에서 특정 "발송 예정" 행에 "제외" 실행 (배너 "해당 발송을 제외 처리했습니다." 확인)
3. 탭 B는 새로고침하지 않은 상태에서 같은 행의 "지금 발송" 클릭 후 confirm [확인]

**기대 결과**
- [ ] 탭 B에 빨간 배너로 409 에러 메시지 "작업 상태가 방금 바뀌었습니다. 새로고침 후 다시 시도하세요." 표시 (상태 사전검사 통과 후 조건부 UPDATE가 0행이면 반환)
- [ ] 발송은 실행되지 않고 잡 상태는 탭 A의 결과(관리자 취소) 그대로 유지

**엣지 케이스**
- [ ] 크론 디스패처가 먼저 잡아 processing으로 바뀐 직후 "지금 발송"을 누르면 단건 claim 실패로 409 "작업이 이미 처리 중입니다." 표시
- [ ] 탭 B에서 같은 행에 "제외"를 시도하면 400 "예정 또는 실패 상태만 제외할 수 있습니다." 표시
- [ ] 같은 예약·단계에 이미 예정된 안내가 있는 상태에서 재발송(retry 경로)을 유발하면 유니크 제약 충돌로 400 "이미 같은 안내가 예정되어 있어 재발송할 수 없습니다." 반환

2. http://localhost:3000/notifications/schedule 접속
3. 개발자도구 콘솔에서 fetch("/api/notifications/jobs", {method:"POST", headers:{"Content-Type":"application/json"}, body: JSON.stringify({jobId:"<임의 발송예정 잡 UUID>", action:"exclude"})}) 실행

- [ ] 행에 "지금 발송" / "시각 변경" / "제외" 버튼이 전혀 표시되지 않음
- [ ] API 직접 호출은 403과 에러 메시지 "이 작업은 관리자만 할 수 있습니다." 반환

**엣지 케이스**

### 9.8 [P2] API 입력 검증·범위 가드 (개발자도구 직접 호출)

**절차**
1. owner로 로그인한 상태에서 개발자도구 콘솔 열기
2. jobId에 UUID가 아닌 값으로 POST /api/notifications/jobs 호출 (예: {jobId:"abc", action:"send_now"})
3. 존재하지 않는 임의 UUID로 호출 (예: {jobId:"00000000-0000-0000-0000-000000000000", action:"send_now"})
4. "발송 예정" 상태 잡에 action:"retry"로 호출

**기대 결과**
- [ ] UUID 형식 오류: zod 검증 실패로 400 반환
- [ ] 존재하지 않거나 다른 사업장의 잡: 404와 "작업을 찾을 수 없습니다." 반환 (business_id 스코프 필터로 타 사업장 잡 접근 차단)
- [ ] scheduled 상태에 retry: 400과 "이 상태에서는 실행할 수 없는 동작입니다." 반환 (retry는 failed·cancelled_by_admin 전용, send_now는 scheduled 전용)

**엣지 케이스**
- [ ] action 값이 enum(send_now/exclude/retry/reschedule) 밖이면 zod 검증으로 400
- [ ] 성공한 액션은 system_audit_logs에 entity_type='notification_job', action, actor_id와 함께 기록됨 (DB에서 확인 가능)

> **참고**: API는 POST /api/notifications/jobs 단일 엔드포인트로 owner 전용(requireUser("owner")). send_now는 dispatchJobById로 해당 jobId 1건만 조건부 UPDATE(status='scheduled'일 때만 processing으로 전환)로 잠금 후 발송하므로, 크론·중복 클릭과 경합해도 한쪽만 성공한다(claimed=0이면 409). 상태 사전검사와 실제 UPDATE 사이의 경합은 .eq("status", job.status) 가드로 0행 갱신 시 409를 반환한다. retry 액션(failed·cancelled_by_admin 대상)은 이 화면이 아닌 실패 관리 화면에서 사용되지만 같은 엔드포인트를 공유하며, 유니크 제약 충돌(23505) 시 400 "이미 같은 안내가 예정되어 있어 재발송할 수 없습니다."를 반환한다. 모든 액션 성공 시 system_audit_logs에 감사 로그가 남는다(entity_type='notification_job', actor_id 기록). fetch 실패 시 화면 공통 폴백 에러 문구는 "처리에 실패했습니다."이다. 파일: D:\thx_halmae\src\app\(dashboard)\notifications\schedule\page.tsx, D:\thx_halmae\src\app\api\notifications\jobs\route.ts

### 10.2 [P0] dispatch 인증 실패 — CRON_SECRET 불일치 시 401 차단

**절차**
1. Authorization 헤더 없이 호출: try { Invoke-RestMethod -Method Post -Uri "http://localhost:3000/api/cron/notifications/dispatch" } catch { $_.Exception.Response.StatusCode.value__ } (PowerShell 5.1은 4xx에서 예외를 던지므로 try/catch로 상태코드 확인)

**기대 결과**
- [ ] 세 경우 모두 HTTP 401, 응답 본문 {"error":"unauthorized"}
- [ ] Job이 전혀 클레임되지 않음 — 발송 일정 화면의 Job 상태(발송 예정) 변화 없음
- [ ] reconcile(/api/cron/notifications/reconcile), recover-locks(/api/cron/notifications/recover-locks)도 동일하게 isCronAuthorized를 쓰므로 무인증 호출 시 각각 401 {"error":"unauthorized"}

**엣지 케이스**
- [ ] 시크릿 앞뒤 공백 포함("Bearer <CRON_SECRET> ") → 길이 불일치로 401 (safeEqual은 길이 다르면 즉시 false)
- [ ] 소문자 "bearer <CRON_SECRET>" → 문자열 불일치로 401

### 10.4 [P1] SOLAPI Webhook 수신 — 시크릿 검증·페이로드 검증·중복 차단

**절차**

2. 같은 body로 한 번 더 호출 (중복 이벤트)
3. 시크릿 없이/틀린 값으로 호출
4. -Body 'not-json' 으로 호출 (JSON 파싱 실패)
5. -Body '[]' 로 호출 (빈 배열 — 스키마 min(1) 위반)
6. [데이터 필요 — 상태 갱신까지 확인하려면] DB에서 기존 notification_deliveries 1건에 provider_message_id='M-TEST-002'를 수동 UPDATE 후, messageId=M-TEST-002·statusCode="4000"으로 호출하고 발송 이력 화면에서 해당 delivery 배지 확인

**기대 결과**
- [ ] 1단계: HTTP 200 {"ok":true} — 매칭되는 delivery가 없으므로 provider_webhook_events에 이벤트만 보관(processed=false)되고 delivery 변화 없음
- [ ] 2단계: HTTP 200 {"ok":true} — event_key(sha256) UNIQUE 위반으로 조용히 건너뜀, 중복 처리 없음
- [ ] 3단계: HTTP 401 {"error":"unauthorized"}
- [ ] 4단계: HTTP 400 {"error":"invalid json"}
- [ ] 5단계: HTTP 400 {"error":"invalid payload"}
- [ ] 6단계: delivery status='delivered', delivered_at 기록, recalculate_notification_job_status로 Job 상태 재계산 → 발송 이력 화면에서 delivery 배지 "수신 완료", "수신 · {시각}" 표시, 이벤트는 processed=true

**엣지 케이스**
- [ ] statusCode가 "4000" 이외(예: "3000") → delivery status='failed', last_error_code=statusCode 저장, 이력 화면에 붉은 글씨 "{에러코드}: {메시지 또는 '알 수 없는 오류'}" 표시

- [ ] 항목 1001개 배열 → 400 invalid payload (max 1000)

(5) recover_stuck_notification_jobs·claim_due_notification_jobs의 정확한 시간 임계값은 마이그레이션 SQL에 정의되어 있어 이번 범위(지정 파일)에서는 확인하지 않았다.

## ⑪ 발송 이력

**경로**: `/notifications/history`

문자 발송 이력(성공·실패·제외·관리자 취소) 목록과 상태 필터, 발송 건별 상세(메시지 본문·외부 메시지 ID·발송/수신 시각·오류) 확인. 조회 전용 화면 — 액션 버튼 없음(재발송은 실패 관리 화면). 데이터는 API 라우트 없이 브라우저 Supabase 클라이언트가 notification_jobs(+reservations, notification_deliveries)를 직접 조회하며 RLS(같은 사업장 구성원 SELECT)가 적용된다. 쿼리는 status IN (success, failed, skipped, cancelled_by_admin), updated_at 내림차순, 최대 100건.

**사전 조건**
- contact@itconnect.dev(owner) 계정으로 http://localhost:3000 로그인 가능해야 함 (Supabase Auth 실계정 + profiles.business_id 필요 — RLS가 이 값으로 필터함)
- 이력 데이터 필요: notification_jobs 상태가 success/failed/skipped/cancelled_by_admin 중 하나인 작업이 최소 1건 있어야 목록이 비지 않음. 없다면 발송 일정(/notifications/schedule) 화면에서 scheduled 작업에 '즉시 발송'을 실행하면 dry_run 모드에서 skipped(사유 dry_run) 이력이 생성됨 — 이를 위해 미래 방문일 예약 + 자동 안내 설정 활성화 + 발송 예정 작업이 있어야 함
- success·failed 상태 이력은 dry_run 모드에서는 UI 조작만으로 생성 불가(발송 모드 게이트가 실제 발송 전에 skipped 처리) — 시나리오 5·필터 검증 일부는 DB 시드(notification_jobs.status='failed'/'success' + notification_deliveries 행) 또는 발송 모드 변경이 필요
- 직원 권한 시나리오는 staff 역할 계정이 별도로 필요 (환경 정보에 staff 계정 미기재 — 없으면 해당 시나리오 보류)

**필요 리소스·환경변수**: `dev 서버 http://localhost:3000 실행 중`, `Supabase(서울) 연결 + 마이그레이션 0005~0007 적용(notification_jobs/deliveries RLS 포함)`, `NOTIFICATION_SEND_MODE=dry_run — dry_run 이력 생성 시나리오의 전제`, `NOTIFICATION_TEST_PHONE_ALLOWLIST 비어있음(allowlist 차단 경로는 이 화면에서 dry_run과 동일하게 '제외됨'으로 보임)`, `PHONE_HASH_PEPPER 설정됨(디스패처가 delivery 생성 시 필요 — 이력 데이터 생성 전제)`

### 11.1 [P0] 이력 목록 기본 로드와 카드 표시

**절차**
1. contact@itconnect.dev로 로그인 후 사이드바 '문자 발송' 그룹에서 '발송 이력' 메뉴 클릭 (NEW 배지 표시)
2. 페이지 로드 직후 화면 상태 관찰
3. 카드 1장의 상단 정보(제목·배지·시각·비용)를 확인

**기대 결과**
- [ ] 로드 중에는 '불러오는 중…' 텍스트가 잠시 표시된 뒤 목록으로 전환
- [ ] 상단에 필터 칩 4개 '전체' / '성공' / '실패' / '제외'가 표시되고 기본 선택은 '전체'
- [ ] 카드 제목이 '{단계} 안내 · {고객명}' 형식 (단계 라벨: D-7, D-3, D-1, 당일, 수동), 옆에 예약 표시번호. 예약 정보가 없으면 '—'
- [ ] 상태 배지: '성공'(녹색), '실패'/'제외됨'/'관리자 취소'(회색). 그 옆에 KST 시각(MM. DD. HH:mm 형식)과 '비용 {n}원' (비용 정보 없으면 '비용 —')
- [ ] 목록은 updated_at 내림차순, 최대 100건

**엣지 케이스**
- [ ] 이력이 4개 대상 상태(success/failed/skipped/cancelled_by_admin) 중 하나도 없으면 '표시할 발송 이력이 없습니다.' 빈 상태 카드 표시
- [ ] notification_deliveries가 0건인 job은 카드 내부에 '발송 기록이 없습니다.' 표시
- [ ] '발송 예정'/'발송 중'/'예약 변경 취소'/'예약 취소' 상태의 작업은 쿼리 대상이 아니므로 이 화면에 절대 나타나지 않아야 함

### 11.2 [P0] dry_run 발송 기록이 이력에 '제외됨'으로 남는지

**절차**
1. (사전) 발송 일정(/notifications/schedule) 화면에서 '발송 예정' 작업 1건에 즉시 발송을 실행하고 '발송 처리를 실행했습니다 — 성공 0건 · 드라이런 1건 · 제외 0건 · 실패 0건' 안내 확인
2. 사이드바 '발송 이력' 클릭
3. 방금 처리한 작업 카드를 찾아 상태 배지와 delivery 행 확인
4. delivery 행의 '자세히 →' 클릭

**기대 결과**
- [ ] 해당 작업이 목록 최상단 부근에 배지 '제외됨'(회색)으로 표시됨 (dry_run은 job status=skipped로 기록)
- [ ] delivery 행: '문자 (SMS)' 또는 '문자 (LMS)' + 회색 배지 '제외'
- [ ] 확장 시 흰 말풍선에 변수가 치환된 실제 메시지 본문이 줄바꿈 유지된 채 표시됨
- [ ] 메타 정보가 '외부 메시지 ID · —' / '발송 · —' / '수신 · —' — dry_run은 실제 발송이 없어 provider_message_id·sent_at·delivered_at 모두 null
- [ ] 빨간 오류 줄이 없음 (오류 줄은 delivery status='failed'이고 last_error_code가 있을 때만 표시되는데 dry_run은 둘 다 아님)
- [ ] 카드 상단에 '비용 {n}원' 표시 (dry_run이라도 estimated_cost는 기록됨)

**엣지 케이스**
- [ ] '성공' 필터를 눌렀을 때 이 dry_run 기록이 절대 나타나지 않아야 함 — '제외' 필터에서만 표시
- [ ] SOLAPI 설정 화면의 상태가 dry_run임을 함께 확인해 실제 문자가 나가지 않았음을 교차 검증

### 11.3 [P0] 상태 필터 칩 4종 동작

**절차**

**기대 결과**
- [ ] '성공': 배지 '성공'인 카드만 표시
- [ ] '실패': 배지 '실패'인 카드만 표시

- [ ] '전체': 4개 상태 카드가 모두 표시
- [ ] 선택된 칩은 활성(on) 스타일로 강조되고, 필터는 클라이언트 측이라 네트워크 재요청이 발생하지 않음

### 11.4 [P1] '자세히' 확장·접기와 단일 확장 동작

### 11.5 [P1] 실패 이력의 오류 코드·메시지 표시 (조회 전용 확인)

## ⑫ 실패 관리

**경로**: `/notifications/failures`

최종 실패한 발송 작업(notification_jobs.status='failed') 목록을 카드로 보여주고, owner가 재발송·발송 제외·처리 완료 액션을 수행할 수 있는지, 마스킹 번호 경고와 빈 상태 표시가 올바른지 검증한다.

**사전 조건**
- dev 서버 http://localhost:3000 실행 중이고 contact@itconnect.dev (owner)로 로그인되어 있을 것
- 사업장 '고마워할매'에 status='failed'인 notification_jobs 데이터가 최소 2건 존재할 것 — 없으면 Supabase에서 직접 시드 필요 (재발송용 1건 + 제외용 1건)
- 마스킹 경고 시나리오용: guest_phone에 '*' 문자가 포함된 예약에 연결된 failed 잡 1건 (엑셀 임포트 마스킹 데이터 등)
- 직원 뷰 시나리오용: staff 권한 계정 1개 (없으면 해당 시나리오 생략)
- NOTIFICATION_SEND_MODE=dry_run 상태 유지 (실발송 방지)

**필요 리소스·환경변수**: `NOTIFICATION_SEND_MODE=dry_run — 재발송 시 실제 문자 대신 드라이런 처리되어 잡이 'skipped'로 전환됨`, `NOTIFICATION_TEST_PHONE_ALLOWLIST 비어있음 — dry_run이므로 allowlist 게이트에는 도달하지 않음`, `Supabase 서울 DB 연결 (notification_jobs, notification_deliveries, reservations, system_audit_logs 테이블)`, `SOLAPI 키·발신번호 설정됨 (dry_run이라 실제 호출 없음)`

### 12.1 [P0] 실패 목록 카드 표시

**절차**

2. 페이지 로딩 중 '불러오는 중…' 문구가 잠깐 표시되는지 관찰한다

**이유**: dispatcher는 발송 시점에 reservation_options를 조회해 #{옵션} 변수를 치환한다. 같은 테이블을 upstream 엑셀 업로드(재업로드 시 delete+insert)와 준비물 매칭 카드가 함께 사용하므로, 옵션 변경이 발송 본문과 예약 상세 준비물 카드 양쪽에 일관되게 반영되는지가 merge 회귀의 핵심 확인 지점이다. 옵션 없는 예약의 '없음' 폴백도 함께 확인한다.

**근거**: `d:\thx_halmae\src\lib\notifications\dispatcher.ts (159~166행: 발송 시점 옵션 조회, '없음' 폴백), d:\thx_halmae\src\app\api\import\revert\route.ts (63~73행 옵션 delete+insert 패턴), d:\thx_halmae\src\components\preparation-card.tsx`

**절차**
1. #{옵션} 변수가 든 템플릿으로 일정 생성
2. 재업로드로 해당 예약의 옵션명을 변경
3. 지금 발송(dry_run) 후 이력 자세히에서 본문의 옵션이 최신 값인지, 예약 상세 준비물 카드도 새 옵션 기준으로 매칭되는지 확인
4. 옵션 없는 예약 발송 → 본문에 '없음' 표기 확인

### C.15 [P2] 사업장 정보 결측 시 변수 제거 — 플레이스홀더 미노출

**영역**: 설정 상호작용: /settings/notifications 사업장 정보 ↔ 발송 본문

**이유**: 사업장전화·주소를 비워둔 채 해당 변수가 든 템플릿을 발송하면 dispatcher가 #{...} 플레이스홀더를 제거하고 missing_vars를 content_snapshot에 기록한다. 자동 안내 설정의 사업장 정보 저장(초안 P1)과 발송 본문 사이의 상호작용이 미검증이다.

**절차**
1. 자동 안내 설정에서 사업장전화·주소를 비우고 저장
2. #{사업장전화}가 든 템플릿의 예정 건을 dry_run 발송
3. 이력 자세히에서 본문에 #{...}가 노출되지 않는지 확인
