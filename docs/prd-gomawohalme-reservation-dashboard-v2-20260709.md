## 1. 제품 개요

고마워할매 예약 운영 대시보드는 고마워할매 대표와 직원이 네이버 스마트플레이스 예약 데이터를 기반으로 예약 현황, 고객 연락처, 방문 일정, 예상 매출, 준비 알림, 준비물 예시, 정산 및 세금계산서 확인 상태를 함께 관리하는 내부 운영 시스템이다.

v1은 고마워할매 1곳 전용 시스템으로 개발한다. 예약 데이터는 엑셀 수동 업로드, 대표 PC 로컬 수집기 실행, 또는 자유 텍스트 문의 입력으로 가져온다. **원본 예약 데이터는 대표 PC의 로컬 엑셀 파일에 마스터로 저장하고(자체 PK 부여), 직원이 공유해서 보는 사본만 Supabase에 반영한다.** 웹 대시보드는 관리자(대표)의 업로드로 갱신되며, 직원은 각자 기기에서 로그인해 갱신된 대시보드를 열람한다. PC와 모바일에서 모두 확인할 수 있다.

---

## 2. 문제 정의

현재 예약 운영은 네이버 스마트플레이스 예약 데이터를 사람이 직접 확인하고 정리하는 방식에 의존한다.

대표와 직원은 예약자, 방문일, 인원, 옵션, 결제금액, 연락처, 예약상태를 반복적으로 확인해야 한다. 이 과정에서 누락, 중복 확인, 준비물 누락, 정산 확인 지연이 발생할 수 있다. 특히 전화·문자로 오는 예약 문의는 네이버 예약번호가 없어 별도로 관리되지 못하고 흩어진다.

특히 예약 데이터가 엑셀과 네이버 화면에 흩어져 있어 다음 정보를 한눈에 보기 어렵다.

- 이번 달 예약이 몇 건인지
- 이번 주 누가 방문하는지
- 오늘 무엇을 준비해야 하는지
- 어떤 예약의 정산과 세금계산서를 확인해야 하는지
- 직원이 어떤 예약을 확인했는지
- 전화·문자로 온 문의가 어떤 상태인지

## 3. 제품 목표

v1의 목표는 예약 데이터를 자동으로 정리하고, 대표와 직원이 같은 기준으로 예약 운영을 확인할 수 있게 만드는 것이다.

| 목표 | 설명 |
|---|---|
| 예약 데이터 통합 | 엑셀·로컬 수집기·텍스트 문의로 가져온 예약을 자체 PK로 통합 관리한다. |
| 로컬 원본 보관 | 원본 예약 데이터는 대표 PC 로컬 엑셀에 마스터로 저장한다. |
| 운영 현황 가시화 | 예약 수, 방문 인원, 예상 매출, 다가오는 예약을 한눈에 보여준다. |
| 직원 공유 | 관리자가 업로드하면 직원이 각자 기기에서 갱신된 대시보드를 열람한다. |
| 예약 준비 지원 | 방문일 기준 캘린더와 준비 알림을 제공한다. |
| 고객 응대 지원 | 예약 상세에서 고객 연락처를 확인할 수 있게 한다. |
| 정산 확인 관리 | 예약별 정산과 세금계산서 확인 상태를 수동 체크한다. |
| 데이터 내보내기 | 원하는 필드만 골라 엑셀로 내보내면서 대시보드에도 동시 반영한다. |
| 모바일 확인 지원 | 현장에서도 주요 예약 정보를 확인할 수 있게 한다. |

## 4. 사용자

### 4.1 대표 / 관리자 (owner)

대표는 전체 예약 운영을 관리하는 유일한 관리자다.

대표는 직원 계정을 초대하거나 비활성화할 수 있고, **엑셀 업로드·로컬 수집기 실행·텍스트 문의 입력·데이터 내보내기 설정** 등 데이터를 바꾸는 모든 작업을 수행한다. 정산·세금계산서 확인, 수집 로그 확인, 전체 예약 현황 조회도 가능하다.

### 4.2 직원 (staff)

직원은 예약 운영 실무를 수행한다.

직원은 예약 목록과 상세 정보를 확인하고, 정산 상태와 세금계산서 상태·메모를 수정할 수 있다. **단, 엑셀 업로드·텍스트 문의 입력·직원 계정 관리·시스템 설정은 할 수 없다**(업로드·관리 메뉴 자체가 보이지 않는다).

## 5. v1 핵심 기능

### 5.1 로그인 및 권한

대표와 직원은 각자 계정으로 로그인한다. 시스템은 사용자를 관리자(owner)와 직원(staff) 역할로 구분한다.

관리자는 전체 기능을, 직원은 열람과 운영 상태 체크만 사용할 수 있다. 권한은 화면 숨김이 아니라 데이터 차원(RLS)에서 강제된다.

### 5.2 직원 계정 관리

관리자는 직원 계정을 초대하거나 비활성화할 수 있다. 비활성화된 직원은 더 이상 시스템에 접근할 수 없다.

### 5.3 엑셀 수동 업로드 (관리자 전용)

관리자는 네이버 스마트플레이스 예약 상세 엑셀 파일을 웹에서 **파일 선택 버튼**으로 업로드할 수 있다. 시스템은 업로드된 파일을 바로 반영하지 않고, 먼저 검증과 미리보기를 제공한다.

### 5.4 업로드 미리보기

시스템은 업로드된 파일에서 신규 예약, 업데이트 예약, 취소 예약, 변경 예약, **병합 후보**, 오류 예약을 구분해 보여준다. 사용자는 미리보기를 확인한 뒤 "반영하기"를 눌러야 실제 예약 데이터가 저장된다.

### 5.5 로컬 수집기 버튼 가져오기 (관리자 전용)

관리자는 PC에 설치된 로컬 수집기를 실행한다. 네이버 스마트플레이스에 로그인된 상태에서 "예약 데이터 가져오기" 버튼을 누르면 예약 데이터를 가져온다. 로컬 수집기는 네이버 아이디와 비밀번호를 저장하지 않는다.

### 5.6 텍스트 예약 문의 입력·검수 (신규, 관리자 전용)

전화·문자·카톡 등으로 온 자유 텍스트 예약 문의를 관리자가 입력하면, 시스템은 이를 `대기` 상태로 저장한다. v1에서는 관리자가 방문일·인원·옵션·이름·연락처를 확인·정리한 뒤 **정식 예약으로 승격**한다. (AI 자동 파싱은 v2에서 도입 예정 — 그 경우에도 관리자 검수 후 승격하는 흐름은 동일하다.) 원문은 항상 보존한다.

### 5.7 예약 데이터 검증

시스템은 예약자명, 연락처, 이용일, 인원, 옵션, 결제금액, 예약상태를 검증한다. 필수 항목이 없으면 저장하지 않고 오류 메시지를 보여준다.

### 5.8 자체 PK 부여 (신규)

모든 예약은 저장 시 **자체 고유 식별자(표시번호 `GMW-260709-001` 형식)**를 부여받는다. 네이버 예약번호는 있으면 함께 저장하고, 없으면(텍스트 문의) 빈 값으로 둔다. 이로써 네이버 예약번호가 없는 문의도 한 기준으로 관리된다.

### 5.9 예약번호 기준 병합

시스템은 같은 예약(네이버 예약번호 또는 자체 PK 기준)으로 나뉜 여러 행을 하나의 예약으로 병합한다. 숙박·바베큐·계곡 옵션이 각각 다른 줄로 들어와도 하나의 예약으로 정리한다.

텍스트 문의로 먼저 만든 예약이 나중에 네이버 엑셀로 유입되면, 시스템은 **이름+연락처+방문일**로 병합 후보를 찾아 미리보기에 표시하고, **관리자가 확인해 병합**한다(신규 중복 생성 대신 연결). 자동 병합은 하지 않는다.

### 5.10 예약 목록

대표와 직원은 예약 목록에서 방문일, 표시번호, 예약자명, 마스킹된 연락처, 인원, 옵션, 결제금액, 예약상태, 정산 상태, 세금계산서 상태를 확인한다.

### 5.11 예약 상세

예약 상세 화면에서는 전체 연락처와 전체 옵션 정보를 확인할 수 있다. 정산 상태, 세금계산서 상태, 메모도 상세에서 수정할 수 있다. **메모는 문자열 텍스트로 저장·표시한다**(객체가 아님 — v1 화면의 `[object Object]` 표시 버그 수정 대상).

### 5.12 연락처 표시 정책

목록에서는 연락처를 마스킹한다(`010-1234-5678` → `010-****-5678`). 상세 화면에서는 전체 연락처를 표시한다.

### 5.13 KPI 영역

대표와 직원은 대시보드 상단에서 이번 달 예약 수, 총 방문 인원, 예상 매출, 다가오는 예약 수, 취소/변경 예약 수를 확인한다.

| KPI | 설명 |
|---|---|
| 이번 달 예약 | **취소 제외(확정+변경) 예약 수** |
| 총 방문 인원 | 취소 제외 예약 기준 인원 합계 |
| 예상 매출 | 취소 제외 예약 결제금액 합계 |
| 다가오는 예약 | 오늘 이후 방문 예정 예약 수 |
| 취소/변경 예약 | 확인이 필요한 예외 예약 수 |

> v1 화면 라벨은 "확정 예약"이었으나 실제 로직은 취소만 제외(변경 포함)이므로, **"취소 제외(확정+변경)"**로 표기를 통일한다.

### 5.14 예약 캘린더

시스템은 방문일 기준으로 예약을 달력에 표시한다. 확정, 변경, 취소 예약은 색상이나 태그로 구분한다.

### 5.15 준비 알림

시스템은 방문일이 가까운 예약을 기준으로 준비해야 할 일을 보여준다. v1에서는 문자나 카카오톡 발송은 하지 않고, 대시보드 안에서만 준비 알림을 표시한다.

| 기준 | 준비 알림 |
|---|---|
| D-7 | 예약 내용 확인 |
| D-5 | 준비물 확인 |
| D-3 | 고객 안내 필요 |
| D-1 | 최종 인원·옵션 확인 |
| D-Day | 오늘 방문 예약 |

### 5.16 준비물 예시 표시

시스템은 예약 옵션을 기준으로 준비물 예시를 표시한다. v1에서는 정확한 수량 계산까지 하지 않는다.

| 예약 옵션 | 준비물 예시 |
|---|---|
| 바베큐 | 고기, 숯, 집게, 장갑, 채소, 일회용 식기 |
| 계곡 | 물놀이 안내, 안전용품, 수건, 구급용품 |
| 버스왕복 | 차량 배차 확인, 탑승 인원 확인, 기사 연락 |
| 매실 | 매실, 설탕, 용기, 장갑 |
| 숙박 | 침구, 객실 정리, 수건, 비품 |

### 5.17 예상 매출

시스템은 취소 제외 예약의 결제금액을 기준으로 월 예상 매출과 예약별 매출을 보여준다. 취소 예약은 매출 계산에서 제외한다.

### 5.18 정산 상태 수동 체크 (Supabase 소유)

대표와 직원은 예약별 정산 상태를 수동으로 체크한다.

| 정산 상태 | 의미 |
|---|---|
| 별도 확인 필요 | 아직 확인하지 않음 |
| 확인 완료 | 정산 확인 완료 |
| 해당 없음 | 취소 또는 정산 대상 아님 |

### 5.19 세금계산서 상태 수동 체크 (Supabase 소유)

| 세금계산서 상태 | 의미 |
|---|---|
| 별도 확인 필요 | 아직 확인하지 않음 |
| 발행 완료 | 세금계산서 발행 완료 |
| 해당 없음 | 발행 대상 아님 |

### 5.20 필드 소유권 및 덮어쓰기 규칙 (신규)

예약 데이터는 소유권이 둘로 나뉜다. **방문일·인원·옵션·금액 등 사실 정보는 로컬 엑셀이 원본**이고, **정산 상태·세금계산서 상태·메모 등 운영 상태는 Supabase가 원본**이다. 엑셀을 재업로드해도 **운영 상태는 절대 덮어쓰지 않는다.** 이로써 직원이 체크한 정산 상태가 대표의 재업로드로 사라지는 사고를 막는다.

### 5.21 기본 상태값

예약이 새로 저장될 때 확정/변경 예약은 정산·세금계산서 상태를 `별도 확인 필요`로 둔다. 취소 예약은 두 상태 모두 `해당 없음`으로 둔다.

### 5.22 수정 이력

정산 상태, 세금계산서 상태, 메모가 변경되면 누가, 언제, 무엇을 바꿨는지 기록한다.

### 5.23 업로드/수집 이력

시스템은 엑셀 업로드, 로컬 수집기 실행, 텍스트 문의 처리 이력을 기록한다. 누가·언제 실행했는지, 몇 건이 반영되었는지, **로컬 엑셀 저장이 성공했는지**, 오류가 있었는지 확인할 수 있다.

### 5.24 마지막 업로드 되돌리기

잘못된 엑셀 파일이 반영되었을 경우, 마지막 업로드 묶음을 되돌릴 수 있다. 과거 특정 시점 전체 복원은 v1 범위에 포함하지 않는다.

### 5.25 데이터 내보내기 + 대시보드 동시 반영 (신규)

관리자가 업로드·반영하면, 시스템은 **① 자체 PK가 부여된 새 엑셀을 대표 PC 로컬에 저장**하고 **② 같은 데이터를 Supabase에 반영해 대시보드를 갱신**하는 두 동작을 하나로 처리한다. 내보낼 때 **원하는 필드만 골라(내보내기 프로필)** 엑셀에 담을 수 있다. 로컬 저장과 Supabase 반영 중 하나라도 실패하면 전체를 되돌리고 재시도를 안내한다.

### 5.26 모바일 화면

대표와 직원은 모바일에서도 오늘 방문 예약, 다가오는 예약, 예약자명, 연락처, 인원, 옵션, 준비 알림, 정산/세금계산서 상태를 확인할 수 있다.

## 6. v1 제외 기능

| 기능 | 제외 이유 |

## 7. 성공 기준

| 성공 기준 | 목표 |
|---|---|
| 예약 데이터 반영 | 엑셀·로컬 수집기·텍스트 문의로 예약이 정상 저장된다. |
| 로컬 원본 보관 | 자체 PK가 부여된 새 엑셀이 대표 PC 로컬에 저장된다. |
| 직원 공유 | 관리자 업로드 후 직원이 각자 기기에서 갱신된 대시보드를 본다. |
| 데이터 정합성 | 같은 예약은 중복 생성되지 않고 업데이트·병합된다. |
| 운영 상태 보존 | 엑셀 재업로드가 정산·세금계산서·메모를 덮어쓰지 않는다. |
| 운영 가시성 | KPI·캘린더·준비 알림으로 이번 달·이번 주 상태를 파악한다. |
| 안전장치 | 미리보기·이력·되돌리기·동기화 롤백으로 잘못된 반영을 줄인다. |
| 모바일 활용 | 현장에서 모바일로 예약 현황을 확인한다. |
