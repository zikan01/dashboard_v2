# 고마워할매 문자 발송 시스템 PRD v3.1

> 문서 유형: Product Requirements Document  
> 작성일: 2026-07-17  
> 대상 저장소: `zikan01/dashboard_v2`  
> 제품 범위: 솔라피(SOLAPI) 기반 문자 발송 시스템 (기존 예약 운영 대시보드와 연동)  
> 주요 기능: 예약 고객 문자 자동 안내, 발송 일정·이력·실패 관리  
> 기준 시간대: `Asia/Seoul`  
> 메시지 제공업체: SOLAPI

---

## 0. 문서 목적

본 문서는 기존 고마워할매 예약 운영 대시보드와 연동하는 문자 발송 시스템의 제품 요구사항을 정의한다.

1. 예약일을 기준으로 고객에게 SMS·LMS 문자를 자동 또는 수동 발송한다.
2. 모든 발송 일정, 성공·실패 결과와 비용을 데이터로 관리한다.

구체적인 화면과 사용자 동작은 FRD, 데이터베이스와 외부 서비스 구조는 TRD에서 정의한다.

v3.0에서 정의했던 카카오 알림톡, 이메일, 고객 사진 전달, 만족도 조사, 네이버 후기 연결 기능은 본 버전 범위에서 제외한다.

## 1. 담당 범위 (협업 경계)

본 시스템은 두 명이 나누어 개발하며, 본 문서는 **문자 발송 시스템 담당 범위만** 정의한다.

| 구분 | 담당 범위 |
|---|---|
| 동료 담당 | 예약 CRUD, 대시보드 홈, 예약 목록, 예약 상세 화면 |
| 본 문서 담당 | 자동안내설정, 메시지 템플릿, 솔라피 설정, 발송 일정, 발송 이력, 실패 관리 6개 페이지와 발송 엔진(스케줄러·SOLAPI 연동·Webhook) |

인터페이스 계약:

- 발송 시스템은 예약 테이블을 **읽기 전용**으로 참조한다. 예약 데이터의 생성·수정은 동료 담당 영역이다.
- 발송 시스템은 예약 상태, 방문일, 전화번호 변경을 감지하여 발송 일정을 재계산한다.
- 동료 담당 화면(예약 상세 등)의 수정이 필요한 기능은 본 범위에서 제외한다. 수동 발송·재발송은 발송 일정과 실패 관리 페이지에서 수행한다.

## 2. 제품 배경

현재 시스템은 네이버 예약 엑셀, 로컬 수집기, 텍스트 예약 문의를 통해 예약 데이터를 통합하고 다음 정보를 관리한다.

- 예약자명과 전화번호
- 방문일과 인원
- 예약 옵션과 결제금액
- 예약 상태
- 정산 상태와 세금계산서 상태
- 운영 메모

하지만 방문 전 안내를 사람이 직접 보내야 한다. 이로 인해 안내 누락, 중복 발송, 변경 전 날짜로의 오발송, 발송 이력 부재가 발생할 수 있다.

---

## 3. 제품 목표

### 3.1 예약 안내 자동화

확정 또는 변경 예약에 대해 다음 기본 일정을 생성한다.

| 시점 | 기본 시간 | 발송 채널 |
|---|---:|---|
| D-7 | 오전 10시 | 문자 |
| D-3 | 오전 10시 | 문자 |
| D-1 | 오후 3시 | 문자 |
| 당일 | 오전 8시 | 문자 |

대표는 각 시점의 활성화 여부, 시간과 템플릿을 변경할 수 있다. SMS·LMS는 최종 치환된 본문 길이에 따라 자동 판별한다.

### 3.2 발송 누락과 중복 방지

- 예약 생성 시 발송 일정을 자동 생성한다.
- 예약일이 변경되면 기존 미발송 작업을 취소하고 새 날짜로 재생성한다.
- 예약이 취소되면 미발송 작업을 모두 취소한다.
- 동일 예약·동일 단계의 중복 발송을 차단한다.
- 발송 직전 최신 예약 상태를 다시 확인한다.

### 3.3 발송 비용과 결과 가시화

- 발송 성공·실패 상태를 기록한다.
- SMS·LMS 유형과 건별 비용을 관리한다.
- 월간 발송량과 비용을 집계한다.

## 5. 주요 사용자

### 5.1 대표·관리자

대표는 다음 작업을 수행한다.

- SOLAPI 발송 설정
- 발신번호 등록 상태와 잔액 확인
- D-7·D-3·D-1·당일 규칙 설정
- 문자 템플릿 관리
- 수동 발송과 실패 재발송
- 발송 이력과 비용 확인

### 5.2 직원

직원은 다음 작업을 수행한다.

- 발송 일정 조회
- 발송 결과 조회

직원은 자동 발송 설정, 템플릿 수정, 수동 발송을 할 수 없다.

### 5.3 예약 고객

고객은 별도 가입이나 동작 없이 예약 안내 문자를 수신한다.

---

## 6. 운영 원칙

### 6.1 Supabase를 운영 데이터의 원본으로 사용

자동 발송에 영향을 주는 다음 정보는 Supabase의 값을 기준으로 한다.

- 고객 전화번호
- 방문일과 예약 상태
- 발송 설정과 일정

로컬 엑셀은 백업, 내보내기와 수동 확인 자료로 사용한다. 엑셀만 수정하고 Supabase에 반영하지 않은 변경은 자동 발송에 적용되지 않는다.

### 6.2 개인정보 최소 수집

발송에 필요한 고객명, 전화번호, 예약 정보만 사용한다. 발송 시스템이 추가로 수집하는 개인정보는 없다.

### 6.3 외부 서비스 장애 대응

SOLAPI가 실패해도 예약과 발송 작업은 보존한다. 실패 원인을 확인하고 수동 재처리할 수 있어야 한다.

---

## 7. 발송 채널

- SMS: 90바이트 이하 단문 (EUC-KR 기준, 한글 45자·영어 90자)
- LMS: 90바이트 초과 장문

최종 치환된 본문의 바이트 수로 SMS·LMS를 자동 판별하고, 유형별 비용을 표시한다.

---

## 8. 자동 발송 대상

다음 조건을 모두 만족하면 자동 발송한다.

- 예약 상태가 `확정(confirmed)` 또는 `변경(changed)`
- 방문일이 존재
- 전화번호가 존재하고 유효함
- 자동 발송이 활성화됨
- 같은 안내가 이미 성공하지 않음
- 발송 시점이 현재 이후이거나 발송 대기 상태임

다음은 발송하지 않는다.

- 취소 예약
- 과거 방문 예약
- 전화번호가 없거나 유효하지 않은 예약
- 관리자가 제외한 예약
- 테스트·중복 예약
- 이미 같은 단계로 성공한 예약

---

## 9. 발송 일정과 변경 규칙

### 9.1 신규 예약

예약 저장 시 활성화된 규칙을 기준으로 작업을 생성한다. 이미 지난 D-7이나 D-3 일정은 생성하지 않는다.

### 9.2 예약 변경

방문일이 바뀌면:

1. 미발송 기존 작업을 `예약 변경 취소`로 변경한다.
2. 새 날짜를 기준으로 작업을 재생성한다.
3. 이미 발송된 이력은 보존한다.
4. 변경 내용을 감사 로그에 기록한다.

### 9.3 예약 취소

예약이 취소되면 모든 미발송 작업을 `예약 취소` 상태로 변경한다. 취소 안내 자동 발송은 v3.1 범위에서 제외한다.

## 10. 템플릿과 미리보기

SMS·LMS 문자 문구를 템플릿으로 관리한다.

지원 변수:

- 고객명
- 방문일과 시간
- 인원
- 옵션
- 예약번호와 표시번호
- 사업장명·전화번호·주소

발송 전 고객 정보, 최종 치환 문구, 예상 문자 유형(SMS·LMS), 예상 비용을 미리 보여준다.

## 12. 수동 발송과 재시도

대표는 발송 일정 화면에서 예정 작업을 즉시 발송할 수 있고, 실패 관리 화면에서 실패 작업을 처리할 수 있다.

- 같은 템플릿으로 재발송
- 연락처 수정 후 재발송 (연락처 수정 자체는 동료 담당 화면에서 수행)
- 발송 제외 처리

중복 가능성이 있으면 경고한다.

## 14. 비용 안내

문자 발송에는 건별 비용이 발생한다.

SOLAPI 표준 단가 (2026-07 요금 페이지 기준, VAT 미포함):

| 유형 | 단가 |
|---|---:|
| SMS | 18원 |
| LMS | 45원 |

월 발송량 구간에 따라 할인되며 통신사 정책에 따라 변동될 수 있다.
단가는 코드에 하드코딩하지 않고 설정값으로 관리한다.

다음 위치에서 안내한다.

- SOLAPI 연결
- 자동 안내 활성화
- 발송 미리보기
- 수동 발송
- 월간 사용량 확인

## 15. 개인정보와 보관

- 화면(목록·상세)에서는 전화번호를 마스킹 없이 전체 표시한다 (2026-07-17 운영 결정).
- 로그와 발송 기록 스냅샷에는 전체 번호를 남기지 않는다 (화면 표시와 별개의 보안 정책).
- 발송 이력에는 발송 시점의 최종 메시지를 보존한다.
- 마케팅 메시지는 본 시스템에서 발송하지 않는다.

## 17. 출시 단계

### 1단계: Dry Run

실제 외부 발송 없이 일정, 템플릿과 예상 비용만 기록한다.

### 2단계: Allowlist

대표와 직원의 테스트 번호에만 실제 발송한다.

### 3단계: 일부 예약 파일럿

일부 고객에게만 운영 발송하여 실패율과 비용을 검증한다.

### 4단계: 전체 운영

확정·변경 예약의 자동 안내를 활성화한다.

---

## 18. 최종 확정 결정

1. 메시지 사업자는 SOLAPI다.
2. 발송 채널은 SMS·LMS 문자만 사용한다. 알림톡·이메일은 범위에서 제외하되 향후 추가 가능한 구조로 둔다.
3. D-7 오전 10시, D-3 오전 10시, D-1 오후 3시, 당일 오전 8시를 기본값으로 한다.
4. 확정·변경 예약만 발송하고 취소 예약은 발송하지 않는다.
5. 자동 발송 운영 데이터는 Supabase를 원본으로 한다.
6. 발송 시스템은 예약 테이블을 읽기 전용으로 참조하며, 예약 CRUD와 관련 화면은 동료 담당이다.
7. 수동 발송·재발송은 발송 일정과 실패 관리 페이지에서 수행한다.
